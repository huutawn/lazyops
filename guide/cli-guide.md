# CLI Guide

## 1. Role

LazyOps CLI is the local operator tool. It is not the primary deploy trigger. The primary trigger is GitHub `push` or `pull_request` after onboarding is complete.

CLI exists to:

- log in once and keep a long-lived CLI identity
- initialize the repository into a valid LazyOps deploy contract
- link the local repo to a LazyOps project
- select or create deployment bindings
- inspect logs, traces, and rollout status
- open optional debug tunnels

## 2. Core Principles

- CLI should feel like Git: login once, use many times
- CLI should stay thin: no local build pipeline, no hidden deploy engine
- CLI should generate logical configuration, not infrastructure secrets
- CLI should work for `standalone`, `distributed-mesh`, and `distributed-k3s`
- CLI should never force the user to classify services as frontend or backend

## 3. Authentication

### 3.1 `lazyops login`

Supported methods:

- email and password
- browser-based Google OAuth2
- browser-based GitHub OAuth2

Expected flow:

1. CLI starts auth.
2. User logs in through browser or direct credentials.
3. Backend returns a PAT from `POST /api/v1/auth/cli-login`.
4. CLI stores PAT in OS keychain if possible, otherwise a protected local credentials file.

### 3.2 `lazyops logout`

- revoke local session
- optionally call PAT revoke endpoint
- remove local stored credentials

## 4. What `init` Must Do

`lazyops init` is the key onboarding command.

It must:

1. scan the current repository
2. detect services from common signals such as `package.json`, `go.mod`, `requirements.txt`, `Dockerfile`, and monorepo structure
3. treat everything as `service`, not `frontend` vs `backend`
4. fetch available deployment targets from backend:
   - `instances`
   - `mesh networks`
   - `clusters`
5. let the user pick the runtime mode:
   - `standalone`
   - `distributed-mesh`
   - `distributed-k3s`
6. create or attach a `DeploymentBinding`
7. generate `lazyops.yaml`

## 5. What `init` Must Not Do

- it must not store SSH credentials
- it must not write raw kubeconfig
- it must not hard-code server IPs as the source of truth
- it must not require the user to write infrastructure YAML

## 6. `lazyops.yaml` as Deploy Contract

The file generated by `init` should contain logical deploy intent only.

Minimum fields:

```yaml
project_slug: acme-shop
runtime_mode: distributed-mesh
deployment_binding:
  target_ref: prod-ap
services:
  - name: web
    path: apps/web
  - name: api
    path: apps/api
compatibility_policy:
  localhost_rescue: true
  env_injection: true
magic_domain_policy:
  provider: sslip.io
scale_to_zero_policy:
  enabled: false
```

Why this works:

- builder reads the file
- backend resolves `target_ref`
- actual deploy targets remain inside backend state
- no SSH data is needed in the repository

## 7. Main Commands

### 7.1 Project setup

- `lazyops login`
- `lazyops init`
- `lazyops link`
- `lazyops doctor`

### 7.2 Observe and operate

- `lazyops logs <service>`
- `lazyops traces <correlation-id>`
- `lazyops status`
- `lazyops bindings`

### 7.3 Optional debug tools

- `lazyops tunnel db`
- `lazyops tunnel tcp`

These are debug features, not the main deployment flow.

## 8. `lazyops link`

`lazyops link` connects the local repository to:

- a LazyOps project
- a GitHub repository that was granted through GitHub App
- an existing or newly created deployment binding

The command should verify:

- repo ownership
- project ownership
- GitHub App installation exists
- selected target is online or at least registered

## 9. `lazyops doctor`

This command should validate:

- CLI auth is valid
- repository is linked
- `lazyops.yaml` is valid
- target binding still exists
- required services and dependencies are declared
- webhook integration looks healthy

## 10. UX Expectations

- any request taking more than one second should show spinner feedback
- config review should be clear before writing `lazyops.yaml`
- destructive actions should require confirmation
- error messages should tell the user what to fix next

## 11. Parallel Work Plan

### 11.1 CLI tracks

- auth and keychain storage
- repository scanner and service detector
- target selection and binding creation
- `lazyops.yaml` generator and validator
- logs and trace commands
- optional tunnel commands

### 11.2 Backend contracts CLI needs early

- auth routes
- project list routes
- instances, mesh, and cluster list routes
- deployment binding create and list routes
- `lazyops.yaml` validation route or schema package
- logs and trace query contracts

## 12. Example Onboarding Flow

1. User runs `lazyops login`.
2. User runs `lazyops init`.
3. CLI scans repo and finds `web`, `api`, and `worker`.
4. CLI fetches available targets.
5. User chooses `distributed-mesh`.
6. CLI creates `DeploymentBinding` with target ref `prod-mesh-ap`.
7. CLI writes `lazyops.yaml`.
8. User links repo to GitHub App installation.
9. User pushes code.
10. LazyOps builds with Nixpacks and deploys through backend-managed target resolution.
